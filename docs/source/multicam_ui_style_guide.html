Auto-Edit UI guide.

A design system for the interview auto-edit platform, built on the Voices brand. Editorial restraint, a dark editing surface, and a small, confident palette. This document is the single reference for anyone — human or AI — designing screens for the tool.

01

Foundations

Colour, type, spacing — the inherited Voices core, extended for a product surface.

The marketing site lives in the light (Parchment). The application lives in the dark — long editing sessions, video-first, where a bright background fatigues the eye and washes out footage. The brand carries across through the same three voices: Ink, Parchment, and Signal Red. Red is rationed: it marks the one thing that matters on a screen, never decoration.

Typography

Three typefaces, unchanged from the brand: DM Serif Display for headings and the wordmark, IBM Plex Sans (Light 300) for everything readable, IBM Plex Mono for labels, timecode, data, and any machine-truth on screen. In a tool full of numbers, the mono face does heavy lifting — every timestamp, threshold, and parameter is mono.

03

Signature pattern — the review player

The heart of the product. Footage first; chrome recedes; data sits beneath the frame.

The player is where the brand earns its keep. The frame is sacred — controls are quiet until needed, the current angle is always labelled, and the timeline below reads like a score: colour-coded lanes you can scan in a second and click to navigate.

00:14:22:08 / 00:41:06:00
● LIVE CUT Voices. B · INTERVIEWEE LUT · clapton_look.cube
Angle
Topics
Energy
Notes

Lane order is fixed: Angle (who's on screen) → Topics (colour = subject) → Energy (waveform) → Notes (markers; red = cut suggestion). Click any block or marker to seek. The white playhead is the only pure-white element on the timeline.

Do
Keep the angle label visible at all times. Let the operator drop back to AUTO from any manual override with one click. Animate angle switches as an instant cut, never a fade — this is an edit tool, cuts are the language.
Don't
Never stack controls over the centre of the frame. Never use red for anything other than presenter, the live-cut marker, cut-suggestion notes, and the single primary action. No spinners over playing video — buffer by holding the current angle.
04

The UI flow

Five screens, one direction of travel: get footage in, let it process, review, refine, export.

The product has a deliberately linear spine. A reviewer should never wonder what to do next — each screen ends by handing off to the next. Processing is the long, unattended middle; everything before it is fast setup, everything after is craft.

01Sign in

TLS gate. Set your reviewer display name — it tags your notes.

→
02Create & ingest

New project, upload three angles, map audio channels to speakers, nudge sync.

→
03Process

Unattended. Sync, proxies, VAD, transcription, AI topics, rough cut. Watch progress or leave.

→
04Review

Player. Watch the auto-cut, switch angles, read topics, leave notes.

→
05Refine & export

Tweak params, request sub-edits, export validated FCPXML to Resolve.

Screen 01 · Sign in

Single centred card on Ink, wordmark above. One password field (or username + password), one display-name field. Calm and bare — this is the only light-touch screen. Failed attempts shake the card and show a mono error line; no detail leaked.

Screen 02 · Create & ingest

A two-step setup that fits one screen. Left: project name and frame rate. Right: three upload dropzones (A / B / C) showing chunked-upload progress bars in Signal Red. Once probed, an audio-channel mapping panel appears — for each detected channel, a speaker dropdown and a ±ms sync nudge. A clapper hint helps. The primary button — "Start processing" — stays disabled until all three angles are mapped.

Screen 03 · Processing

An honest progress screen — no fake spinners. Each sub-job is a row with its own state badge and a one-line plain-English message ("Transcribing interviewee channel — 6m of 41m"). The operator can close the tab; work continues. When everything is green, the screen turns over to a single primary button: "Open in player". Errors surface inline with a Retry on the failed sub-job only.

Screen 04 · Review

The player (Section 03) full-bleed, with a collapsible right rail: notes list (grouped by reviewer), topic index (click to jump), and the conciseness report. The rail never covers the frame — it pushes the player left. This is where reviewers spend their time, so it carries the most polish and the least chrome.

Screen 05 · Refine & export

Two halves. Left, the auto-edit parameters as labelled mono controls (min_shot, lead-in, overlap→wide, periodic wide) with a "Regenerate" that re-cuts live. Below them, a natural-language box — "1 minute on the immigration topic" — that produces a new sub-edit as its own version. Right, the cut versions list (rough / themed / social), each with an Export button that runs the CDL validator first and only then writes FCPXML. A blocked export shows the exact failing clip, never a blank file.

Do
Make every parameter a labelled mono control with its live value shown. Keep "Regenerate" reversible — versions are cheap, nothing is destructive. Validate before export and explain any failure in one sentence.
Don't
Don't let export run on an unvalidated CDL. Don't hide which source files an export references. Don't bake the LUT into anything — it is preview-only.
05

Interface voice

How the product speaks. Direct, spare, grounded — the brand voice, applied to buttons and errors.

Carried straight from the Voices brand: plain words, no hype, no cheerleading. The interface tells the truth about what's happening and what it needs. Labels are verbs. Errors say what broke and what to do.

On voice
"Start processing." · "FCPXML couldn't be written — clip 38 isn't on a frame boundary." · "Transcribing interviewee — 6m of 41m." · "Drop three angles to begin." · "Back to auto."
Off voice
"Let's get started! 🎬" · "Oops! Something went wrong." · "Your amazing edit is being magically created…" · "Awesome — upload complete!" · "Click here to continue your journey."

Microcopy rules: timecode and parameters are always mono. Numbers are never spelled out. Progress is concrete (units, counts), never a percentage alone. Buttons name the action that happens on click. Empty states use one DM Serif line and one quiet instruction.
